## 새 설계 문서 작성 가이드

### 파일 명명 규칙
```
YYYY-MM-DD_{기능명}-design.md
예: 2025-01-15_kanban-board-design.md
```

### 설계 문서 필수 포함 항목

```markdown
## 관련 기획 문서
링크: plan-docs/{파일명}

## 데이터 모델 변경
새 테이블 또는 컬럼 추가 여부, ERD

## API 설계
새 엔드포인트 또는 변경되는 엔드포인트 명세

## 컴포넌트 설계
새 컴포넌트 또는 변경되는 컴포넌트

## 화면 설계
주요 화면 와이어프레임 또는 설명

## 상태 관리
새로운 상태 플로우 (있는 경우)

## 기술 결정 사항
왜 이 방식을 선택했는가 (트레이드오프 포함)

## 영향 범위
변경으로 인해 영향받는 기존 기능
```

## 설계 원칙 (현재 프로젝트)

### UI
- 모든 스타일 inline CSS (`#080809` 배경, `#c8a96e` 골드 액센트)
- 글라스모피즘: `rgba(255,255,255,0.03~0.05)` 배경, `rgba(255,255,255,0.06~0.08)` 보더
- 공통 컴포넌트 최대한 재사용 (`Button`, `Card`, `PageHeader`, `StatusBadge`)

### API
- RESTful 설계, 표준 HTTP 동사 사용
- 목록 응답: `{ data: [...], total: N }` 형식 유지
- 모든 라우트에 `requireAuth` 미들웨어

### DB
- 새 테이블: `supabase/schema_content_ops.sql` 에 마이그레이션 추가
- `user_id` 컬럼 필수 (RLS 정책 기반)
- `created_at`, `updated_at` 타임스탬프 표준 포함
